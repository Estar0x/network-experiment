# 基于 UDP 的远程终端实验

本项目实现了指导书中的 UDP 远程终端：自定义固定头部协议、回退 N 步 ARQ、滑动窗口发送、接收窗口通告流量控制、多客户端、心跳保活，以及基于 PTY 的远程 shell。

## 文件结构

| 文件 | 说明 |
| --- | --- |
| `udp_terminal/protocol.py` | 自定义协议封包/解包、Go-Back-N 发送器、按序接收器、接收窗口通告 |
| `server.py` | UDP 服务端，为每个客户端 ID 创建独立 PTY shell |
| `client.py` | UDP 客户端，原始终端模式转发按键并显示远端输出 |

## 协议格式

固定头部共 13 字节：

| 字段 | 长度 | 说明 |
| --- | --- | --- |
| 魔数 | 2 字节 | 固定 `0x5554` |
| 报文类型 | 1 字节 | `0x01` 输入、`0x02` 输出、`0x03` ACK、`0x04` 心跳 |
| 序列号 | 4 字节 | 用于 ACK、重传、去重 |
| 数据长度 | 2 字节 | 数据段字节数 |
| 客户端 ID | 4 字节 | 区分不同客户端 |

数据段继续用 1 字节子类型区分输入、窗口大小、关闭、输出数据等业务语义。ACK 报文的头部序列号表示累计确认号，ACK 数据段携带 2 字节接收窗口，表示接收端剩余缓冲包数。UDP 单包控制在 1472 字节以内，避免 IP 分片。

## 核心机制：可靠传输与流量控制

本项目在 UDP 之上实现了应用层可靠传输。UDP 只负责发送数据报，不保证到达、顺序和去重，因此协议中加入了序列号、累计 ACK、超时重传、滑动窗口和接收窗口通告。

### 1. 整体分层

本实验没有修改 UDP 协议本身，而是在 UDP 的数据部分设计了一层自定义应用层协议。整体可以分成三层理解：

```text
应用层：UDP 远程终端协议
  - 自定义协议头
  - 命令输入、命令输出、ACK、心跳
  - 序列号、客户端 ID、滑动窗口、流量控制

传输层：UDP
  - 使用 socket.SOCK_DGRAM 发送和接收数据报
  - 不保证可靠性、不保证顺序、不自动重传

网络层：IP
  - 负责在主机之间转发数据包
```

因此，本项目的核心思想是：利用 UDP 完成基本数据报收发，再在应用层补上可靠传输、顺序控制、多客户端区分和在线检测等能力。

### 4. 为什么需要 ACK

UDP 调用 `sendto()` 成功，只能说明数据交给了本机协议栈，不能说明对方一定收到了。因此发送方需要接收方返回 ACK，才能确认某个报文已经到达。

基本过程如下：

```text
发送方发送 seq=1 的可靠报文
接收方收到后返回 ACK 1
发送方收到 ACK 1 后，认为 seq=1 已成功送达
```

如果发送方在超时时间内没有收到 ACK，就认为报文或 ACK 可能丢失，于是进行重传。

```text
发送 seq=1
等待 ACK 1
超时未收到 ACK 1
重新发送 seq=1
```

ACK 同时还能用于去重。假如接收方已经处理过 `seq=1`，但 ACK 在网络中丢失，发送方会重传 `seq=1`。接收方再次收到后，可以根据序列号判断这是重复报文，不再重复交给业务层执行，只需要重新发送 ACK。

### 5. 回退 N 步 ARQ

回退 N 步 ARQ，也叫 Go-Back-N ARQ，用来解决 UDP 丢包、乱序和重复的问题。发送端给每个可靠报文分配递增序列号，接收端只接受当前期望的序号。

例如接收端正在等待 `seq=2`，如果先收到了 `seq=3`，说明中间的 `seq=2` 可能丢了。此时接收端不会缓存 `seq=3`，而是丢弃它，并重复确认最后一个连续收到的报文。

ACK 采用累计确认：

```text
ACK 3 表示 seq <= 3 的报文都已经连续正确收到。
```

如果发送端超过 `ack_timeout` 仍没有收到新的累计 ACK，就认为最早的未确认报文超时，然后重传整个未确认窗口。这就是“回退 N 步”的含义。

```text
已发送：seq=1 seq=2 seq=3 seq=4
如果 seq=2 丢失，发送端超时后重传：seq=2 seq=3 seq=4
```

代码中，`GoBackNSender` 负责发送、等待 ACK 和超时重传；`GoBackNReceiver` 负责按序接收、丢弃乱序报文并返回累计 ACK。

### 6. 滑动窗口发送

停等 ARQ 每次只能发送一个报文，必须等 ACK 回来后才能发送下一个，效率较低。滑动窗口允许发送端一次发送多个尚未确认的报文。

例如窗口大小为 `4` 时，发送端可以连续发送：

```text
seq=1 seq=2 seq=3 seq=4
```

此时这 4 个报文都处于“已发送但未确认”状态。收到 ACK 后，窗口向右滑动，新的报文才能进入窗口继续发送。

```text
收到 ACK 1 前：窗口覆盖 seq=1 seq=2 seq=3 seq=4
收到 ACK 1 后：窗口滑动到 seq=2 seq=3 seq=4 seq=5
收到 ACK 3 后：窗口滑动到 seq=4 seq=5 seq=6 seq=7
```

在实现中，`_queue` 保存尚未发送的报文，`_outstanding` 保存已经发送但尚未确认的报文。发送线程会在窗口未满时不断从 `_queue` 取出报文发送。

### 7. 滑动窗口流量控制

滑动窗口解决的是发送效率，但还需要防止发送方过快发送导致接收方缓冲区被占满。因此 ACK 中除了累计确认号，还携带接收端剩余缓冲区大小，也就是通告窗口。

发送端实际可用窗口为：

```text
实际发送窗口 = min(本地发送窗口, 对端通告窗口)
```

例如本地窗口大小为 `8`，但对端 ACK 中通告窗口为 `3`，那么发送端最多只能同时发送 3 个未确认报文。这样可以让发送速度跟随接收端处理能力变化。

当接收端缓冲区释放空间时，会再次发送 ACK 通告新的剩余窗口，发送端收到后继续按新的通告窗口发送。

注意这里实现的是流量控制，不是拥塞控制。流量控制关注“接收方是否处理得过来”，拥塞控制关注“网络链路是否拥塞”。本实验只要求并实现了滑动窗口流量控制。

## 运行方法

服务端运行在 Linux/macOS 主机：

```bash
python3 server.py --host 0.0.0.0 --port 9000 --verbose
```

可选参数：

```bash
python3 server.py --port 9000 --window-size 8 --recv-buffer 16 --ack-timeout 1.0 --retries 5
```

客户端连接：

```bash
python3 client.py 服务器IP 9000
```

客户端也可设置窗口和缓冲区：

```bash
python3 client.py 服务器IP 9000 --window-size 8 --recv-buffer 16
```

客户端进入原始终端模式后，所有按键会透传给服务端 PTY。退出客户端按 `Ctrl+]`；在远程 shell 里执行 `exit` 也会关闭会话。

非交互测试模式：

```bash
python3 client.py 127.0.0.1 9000 --command "pwd"
python3 client.py 127.0.0.1 9000 --command "ls -la"
```
