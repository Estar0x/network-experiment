# 基于 UDP 的远程终端程序设计与实现实验报告

## 一、实验目的

本实验实现一个基于 UDP 的远程终端系统，重点掌握 UDP 无连接、不可靠、面向报文的特点，并在应用层设计协议，实现回退 N 步 ARQ、滑动窗口发送、接收窗口通告流量控制、多客户端隔离和心跳保活。

## 二、总体设计

系统由客户端、服务端和协议模块组成。客户端负责采集本地终端输入，将按键字节封装为 UDP 协议报文发送给服务端；服务端根据客户端 ID 维护独立会话，为每个客户端创建一个 PTY shell，并将远端输出实时封装回传。客户端和服务端都使用同一套回退 N 步可靠传输机制：发送端维护滑动窗口，接收端按序交付并返回累计 ACK。

## 三、协议设计

固定头部包含魔数、报文类型、序列号、数据长度和客户端 ID，头部长度为 13 字节。魔数固定为 `0x5554`，非法魔数、长度不一致或载荷超限的报文会被直接丢弃。

报文类型包括：

| 类型 | 数值 | 说明 |
| --- | --- | --- |
| 命令/输入 | `0x01` | 客户端发送终端输入、窗口大小或关闭请求 |
| 输出 | `0x02` | 服务端返回远程终端输出 |
| ACK | `0x03` | 确认已收到指定序列号报文 |
| 心跳 | `0x04` | 客户端定时保活 |

## 四、可靠传输设计

系统采用回退 N 步 ARQ。发送方维护发送窗口，窗口内允许连续发送多个尚未确认的报文。接收方只接受当前期望序号的报文，并将其放入接收缓冲区等待业务层处理；如果收到重复报文或乱序报文，接收方不会交付给业务层，而是重复返回最近一次累计 ACK。

ACK 报文的头部序列号表示累计确认号，即接收方已经连续正确接收到的最大序号。若发送方等待最早未确认报文超时，则从该报文开始回退，重传当前所有未确认报文。超过最大重传次数后，发送方认为对端不可达并提示异常。

系统同时实现滑动窗口流量控制。ACK 数据段携带接收端剩余缓冲包数，发送方实际可发送窗口为本地窗口和对端通告窗口的较小值。当接收端缓冲区满时，通告窗口变为 0，发送方暂停发送新报文；当接收端释放缓冲空间时，会发送窗口更新 ACK，使发送方继续发送。

为避免窗口更新 ACK 丢失导致发送方长期停在 0 窗口状态，系统加入了零窗口探测：当发送方仍有待发送数据、当前没有未确认报文且对端通告窗口为 0 时，发送方允许发送 1 个探测报文，触发接收方重新返回累计 ACK 和当前剩余窗口。

## 五、关键功能实现

服务端使用客户端 ID 区分用户，每个客户端拥有独立 PTY shell，因此多客户端同时操作不会互相干扰。客户端每 5 秒发送一次心跳，服务端长时间未收到心跳则关闭对应会话并释放资源。

终端控制字符采用透明传输方式处理。客户端进入原始终端模式后，将退格、Tab、Ctrl+C、方向键以及 ANSI 相关字节序列原样发送给服务端 PTY。服务端输出也透明回传到客户端，从而支持实时输出以及 `top`、`vim` 等全屏交互程序的基本使用。

## 六、测试设计

建议测试以下场景：

| 测试项 | 操作 | 预期结果 |
| --- | --- | --- |
| 单客户端命令 | 执行 `pwd`、`ls -la` | 客户端正确显示远程输出 |
| 大输出传输 | 执行 `seq 1 1000` | 多个输出报文可按序显示 |
| 流量控制 | 客户端使用 `--recv-buffer 1` 后执行大输出命令 | 发送端根据通告窗口降速，输出仍完整 |
| 多客户端 | 两个客户端使用不同客户端 ID 同时连接 | 两端 shell 独立，输出不混乱 |
| 心跳保活 | 客户端保持连接 | 服务端持续保留会话 |
| 离线检测 | 直接断开客户端 | 服务端超时后释放会话 |
| Ctrl+C | 执行 `ping 127.0.0.1` 后按 Ctrl+C | 远程命令被中断 |
| 全屏程序 | 执行 `top` 或 `vim` | 界面可以刷新，按正常按键退出 |
| 抓包验证 | 使用 Wireshark 过滤 `udp.port == 9000` | 可观察自定义头部和 ACK 报文 |

## 七、实验结论

本实验在 UDP 之上实现了自定义应用层协议、回退 N 步 ARQ、滑动窗口发送和接收窗口通告流量控制，并结合 PTY 实现了接近真实终端的远程交互。通过序列号、累计 ACK、超时回退重传、心跳和客户端 ID，可以在 UDP 不保证可靠性的前提下完成稳定的远程终端通信。
